# グラウンド空き監視ツール

東京5区の野球場・運動場の空き（キャンセル発生）を20分おきに監視し、
新しい空きが出たら Discord に通知します。GitHub Actions 上で動くので**完全無料・PC起動不要**です。

## 監視対象

| 区 | 施設 | システム |
|---|---|---|
| 中央区 | 月島運動場・浜町運動場 | 11489.jp |
| 品川区 | 天王洲公園・八潮北公園（野球場） | cm9.eprs.jp |
| 大田区 | 大田スタジアム | うぐいすネット |
| 墨田区 | 錦糸公園野球場 | 公共施設予約システム |
| 杉並区 | 上井草スポーツセンター・下高井戸運動場 | さざんかねっと |

条件: **土日祝 / 開始10時〜17時のコマ / 翌月末まで**（`config.yaml` で変更可）

> ⚠️ 大田スタジアムは人工芝改修工事のため **2026/10/1〜2027/1/29 供用停止予定**（区のお知らせより）

## セットアップ手順（初めての方向け）

### 1. Discord の Webhook URL を作る（5分）

1. Discord で通知を受け取りたいサーバーのチャンネルを開く
2. チャンネル名の横の **⚙️（チャンネルの編集）** → **連携サービス** → **ウェブフック**
3. **新しいウェブフック** → 名前を「グラウンド監視」などにする
4. **ウェブフックURLをコピー** を押す（あとで使うのでメモ帳に貼っておく）
   ※このURLは秘密情報です。人に教えないでください

### 2. GitHub アカウントを作る（5分）

1. https://github.com/ を開き **Sign up**
2. メールアドレス・パスワード・ユーザー名を入力して登録
3. 届いた確認メールのコードを入力

### 3. リポジトリ（置き場所）を作る（3分）

1. 画面右上の **＋** → **New repository**
2. Repository name: `ground-monitor`
3. **Public** を選ぶ（重要: Public なら Actions の実行時間が無制限で無料。
   Private だと月2,000分制限があり、20分間隔だと足りなくなります）
4. **Create repository** を押す

### 4. ファイルをアップロードする（10分）

**(a) 普通のファイル**

1. リポジトリ画面の **Add file** → **Upload files**
2. このフォルダの中身（`config.yaml`, `requirements.txt`, `.gitignore`, `monitor` フォルダ, `README.md`）を
   まとめてドラッグ＆ドロップ
3. 下の **Commit changes** を押す

**(b) ワークフローファイル（手で作る必要あり）**

1. **Add file** → **Create new file**
2. ファイル名の欄に `.github/workflows/monitor.yml` と入力（`/` を打つとフォルダになります）
3. このフォルダの `.github/workflows/monitor.yml` の中身をコピーして貼り付け
4. **Commit changes** を押す

### 5. Webhook URL を Secrets に登録する（3分）

1. リポジトリの **Settings** タブ → 左メニュー **Secrets and variables** → **Actions**
2. **New repository secret**
3. Name: `DISCORD_WEBHOOK_URL`　／　Secret: 手順1でコピーしたURL
4. **Add secret**

### 6. 動かしてみる（5分）

1. リポジトリの **Actions** タブを開く
2. 「ワークフローを有効にしますか」と出たら有効化する
3. 左の **ground-monitor** → 右の **Run workflow** → 緑のボタンを押す
4. 1〜2分後に実行が始まり、5分前後で終わります
5. ✅緑のチェックが付けば成功。以後は自動で20分おきに動きます

> **初回は通知が来ません。** 初回は「現在の空き状況」を基準として保存するだけで、
> 2回目以降に**新しく増えた空き**だけを通知します（通知が鳴りっぱなしになるのを防ぐため）

## 失敗したとき（❌赤いバツが付いたとき）

1. Actions タブ → 失敗した実行をクリック → **monitor** をクリックするとログが見えます
2. ページ下部の **Artifacts** に `debug-xxxx` がある場合、その中に
   失敗時のスクリーンショットとHTMLが入っています
3. それを Claude に貼り付けて「ここで失敗してるので直して」と言えば直せます

特に **大田区（うぐいすネット）だけは事前にページ構造を完全確認できていない**ため、
初回はエラーになる可能性があります。その場合は debug 成果物を使って調整してください。
（他の4区は実ページで構造確認済みです）

## 設定変更

`config.yaml` を編集してコミットすれば反映されます。

- 監視する曜日・開始時刻の範囲・何ヶ月先まで見るか → `filters`
- 施設の追加・削除、行の絞り込み → `sites`
- 各区の監視を止める → `enabled: false`
- 監視間隔・稼働時間帯 → `.github/workflows/monitor.yml` の `cron`

## マナー設計（サーバーに優しく）

- 巡回は20分間隔、深夜（JST 0〜7時）は停止
- サイト間に3秒の間隔、ページ遷移ごとに待機を入れています
- User-Agent に監視ボットであることを明示
- 予約サイトの負荷が気になる場合は cron を `*/30` などに広げてください

## 仕組み

```
GitHub Actions (20分おき)
  └─ Playwright(Chromium) で各区の予約サイトを巡回
      └─ 土日祝×10-17時開始の空きコマを抽出
          └─ 前回結果(state.json)と比較して「新しく出た空き」を検出
              └─ Discord Webhook に通知、state.json をコミットして保存
```
